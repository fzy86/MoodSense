§1. API Design (0/10)

Perspective: HTTP semantics, URI naming, response shapes, versioning, pagination, documentation, and error response format.

  • [FAIL] RESTful URL naming (verbs vs nouns, resource naming) (api_001)
    All three endpoints use verb-based naming which violates REST conventions. '/uploadMood' uses the verb 'upload' instead of a noun-based resource path like 'POST /moods'. '/getMoodDistribution' embeds the verb 'get' instead of using 'GET /moods/distribution' or 'GET /mood-distributions'. '/getHappyNearByLocations' similarly embeds 'get' and should be 'GET /locations?mood=HAPPY' or 'GET /locations/nearby'. Additionally, all URLs use camelCase (e.g., 'uploadMood', 'getMoodDistribution') instead of the REST convention of lowercase kebab-case (e.g., 'mood-distributions'). There is no resource hierarchy or nesting — all endpoints are flat at the root path with no common base path like '/api/moods'. The URLs read like RPC-style method names rather than RESTful resource identifiers.
    • src/main/java/com/example/moodsense/demo/controller/MoodSenseController.java
  • [FAIL] Response status codes (proper HTTP status codes returned) (api_003)
    The uploadMood POST endpoint returns void, which defaults to HTTP 200 OK. For a resource creation operation, the correct status code is 201 Created, and ideally the response should include a Location header pointing to the newly created resource. The two GET endpoints always return 200 OK wrapped in ResponseEntity, but there is no handling for error cases: no 404 Not Found when data doesn't exist, no 400 Bad Request responses for invalid input, and no structured error response body. There is no @ControllerAdvice or @ExceptionHandler defined anywhere in the project for centralized exception handling, so validation failures (from @Validated) and other exceptions will produce Spring's default whitelabel error responses with inconsistent formats. The @Validated annotation on @RequestParam fields does not trigger Bean Validation without additional configuration (a @Validated annotation on the class level or a MethodValidationPostProcessor bean).
    • src/main/java/com/example/moodsense/demo/controller/MoodSenseController.java
  • [FAIL] API versioning strategy (api_005)
    There is no API versioning strategy implemented. The endpoints are mounted directly at the root path (e.g., '/uploadMood') with no version prefix such as '/api/v1/moods'. No URI-based versioning ('/v1/...'), header-based versioning (custom 'X-API-Version' or Accept header with version media type), or query parameter-based versioning ('?version=1') is used. The application.properties file does not configure a 'server.servlet.context-path' that could serve as a base path. Without versioning, any breaking changes to the API contract will directly impact existing clients with no migration path. This is a significant concern for API evolution and maintainability.
    • src/main/java/com/example/moodsense/demo/controller/MoodSenseController.java
    • pom.xml
  • [FAIL] Request/Response DTO usage (vs entity exposure) (api_006)
    JPA entities are directly exposed in both the request and response layers, violating the separation of concerns between the API contract and the persistence model. The uploadMood endpoint accepts the Mood @Entity directly as a form-backed model attribute, which exposes internal fields like 'id' (auto-generated) and 'person' (set from the authenticated user) to client manipulation. The getHappyNearByLocations endpoint returns List where Location is a JPA @Entity, leaking persistence annotations and structure to API consumers. While MoodDistribution is a Spring Data projection interface (not an entity), it is also not a proper DTO — it is tightly coupled to the repository query. No dedicated request DTOs (for input validation and transformation) or response DTOs (for controlling the API shape) exist anywhere in the project. This creates tight coupling between the database schema and the API contract, making independent evolution of either impossible.
    • src/main/java/com/example/moodsense/demo/controller/MoodSenseController.java
    • src/main/java/com/example/moodsense/demo/model/Mood.java
    • src/main/java/com/example/moodsense/demo/model/Location.java
    • src/main/java/com/example/moodsense/demo/model/MoodDistribution.java
  • [FAIL] Pagination support for list endpoints (api_007)
    Neither of the two GET endpoints supports pagination. '/getMoodDistribution' returns a raw List and '/getHappyNearByLocations' returns a raw List — both without any page, size, or sort parameters. The MoodRepository extends CrudRepository instead of PagingAndSortingRepository or JpaRepository, so the repository layer also lacks built-in pagination support. For production use, unbounded list responses are a performance and scalability risk: as data grows, these endpoints will return increasingly large payloads with no way for clients to paginate or limit results. Spring Data's Pageable/Page abstractions are not used anywhere in the codebase.
    • src/main/java/com/example/moodsense/demo/controller/MoodSenseController.java
    • src/main/java/com/example/moodsense/demo/repositary/MoodRepository.java
  • [FAIL] HATEOAS / Hypermedia support (api_008)
    There is no HATEOAS or hypermedia support. The 'spring-boot-starter-hateoas' dependency is not included in pom.xml. No Spring HATEOAS classes are used — no EntityModel, CollectionModel, RepresentationModel, or WebMvcLinkBuilder references exist. API responses are plain JSON arrays/objects with no '_links' or '_embedded' sections. Clients must hardcode endpoint URLs rather than discovering them through hypermedia navigation. While HATEOAS is not always required (especially for simple internal APIs), its complete absence means the API achieves Level 2 on the Richardson Maturity Model at best and does not qualify as a fully RESTful API.
    • pom.xml
    • src/main/java/com/example/moodsense/demo/controller/MoodSenseController.java
  • [FAIL] API documentation (Swagger/OpenAPI) (api_009)
    There is no API documentation tooling configured. The pom.xml does not include 'springdoc-openapi-ui' (for OpenAPI 3.0) or 'springfox-swagger2'/'springfox-boot-starter' (for Swagger 2). No @Operation, @ApiResponse, @Parameter, @Schema, or @Tag annotations are present on any controller method or model class. There is no Swagger UI available, no OpenAPI specification endpoint ('/v3/api-docs' or '/swagger-ui.html'), and no static API documentation files. Without machine-readable API documentation, consumers must rely entirely on reading source code to understand the API contract, request/response formats, authentication requirements, and error scenarios.
    • pom.xml
    • src/main/java/com/example/moodsense/demo/controller/MoodSenseController.java
  • [FAIL] Consistent endpoint naming patterns (api_010)
    Endpoint naming is inconsistent across the three routes. '/uploadMood' uses 'upload' as its verb while '/getMoodDistribution' and '/getHappyNearByLocations' use 'get', showing inconsistent verb choice. The casing is inconsistent: 'NearBy' in '/getHappyNearByLocations' capitalizes 'By' while the rest uses standard camelCase. The naming granularity varies: 'uploadMood' is short and generic, 'getMoodDistribution' is medium-length, and 'getHappyNearByLocations' is long and encodes business logic (the 'Happy' filter) directly into the URL. There is no common base path grouping these endpoints under a shared resource prefix (e.g., '/api/moods'). The controller class itself lacks a @RequestMapping at the class level to establish a base path. A consistent pattern would use noun-based resources under a common prefix: 'POST /api/v1/moods', 'GET /api/v1/moods/distribution', 'GET /api/v1/locations/nearby?mood=HAPPY'.
    • src/main/java/com/example/moodsense/demo/controller/MoodSenseController.java
  • [WARN] HTTP method usage (GET/POST/PUT/DELETE appropriateness) (api_002)
    The HTTP methods used are partially correct: POST is used for creating a mood entry ('/uploadMood') and GET is used for the two read-only query endpoints. However, the API is incomplete — there are no PUT or PATCH endpoints for updating an existing mood, and no DELETE endpoint for removing moods. This means the API does not support full CRUD operations on the Mood resource. Additionally, the controller uses the legacy '@RequestMapping(method = RequestMethod.POST)' syntax instead of the more modern and idiomatic '@PostMapping' and '@GetMapping' shortcut annotations available since Spring 4.3. While not technically incorrect, this makes intent less clear at a glance.
    • src/main/java/com/example/moodsense/demo/controller/MoodSenseController.java
  • [WARN] Content type / media type handling (api_004)
    No explicit 'produces' or 'consumes' media type declarations are present on any @RequestMapping annotation. While Spring's @RestController defaults to 'application/json' via Jackson auto-configuration, best practice is to explicitly declare 'produces = MediaType.APPLICATION_JSON_VALUE' so the API contract is self-documenting and the server properly rejects requests with unsupported Accept headers. More critically, the uploadMood endpoint accepts its data as @RequestParam query/form parameters rather than a @RequestBody JSON payload. This means the POST endpoint effectively expects 'application/x-www-form-urlencoded' data rather than 'application/json', which is inconsistent with modern REST API conventions. The Mood object is bound as a form-backed model attribute (no @RequestBody annotation), further mixing form-based and REST paradigms.
    • src/main/java/com/example/moodsense/demo/controller/MoodSenseController.java

§2. Domain Structure (2/10)

Perspective: Controller/service/repository separation, design patterns, module boundaries, dependency management, configuration, and startup behaviour.

  • [FAIL] Package naming convention (dom_001)
    The persistence package is named `repositary` instead of the standard `repository`, which breaks Java naming expectations and will confuse maintainers and tooling.
    • src/main/java/com/example/moodsense/demo/repositary/MoodRepository.java
    • src/main/java/com/example/moodsense/demo/service/MoodServiceImplementation.java
    • src/test/java/com/example/moodsense/demo/service/MoodServiceImplementationTest.java
  • [FAIL] Entity design (dom_004)
    JPA entities lack audit fields (`createdAt` / `updatedAt`). `Mood` uses primitive `int` for `@GeneratedValue` id while `MoodRepository` extends `CrudRepository`, so the declared ID type does not match the entity. Lombok `@Data` on `@Entity` types is risky for `equals`/`hashCode` (typically should exclude the id or follow a JPA-safe pattern).
    • src/main/java/com/example/moodsense/demo/model/Mood.java
    • src/main/java/com/example/moodsense/demo/model/Location.java
    • src/main/java/com/example/moodsense/demo/repositary/MoodRepository.java
  • [FAIL] DTO separation (dom_006)
    The REST controller accepts and mutates the `Mood` JPA entity directly (`uploadMood(..., Mood mood, ...)`), coupling the HTTP API to the persistence model instead of using request/response DTOs.
    • src/main/java/com/example/moodsense/demo/controller/MoodSenseController.java
    • src/main/java/com/example/moodsense/demo/model/Mood.java
  • [FAIL] Application entry point (dom_010)
    The Spring Boot main class is named `DemoApplication` rather than something aligned with the product (for example `MoodSenseApplication`), which obscures the real application identity in logs and documentation.
    • src/main/java/com/example/moodsense/demo/DemoApplication.java
  • [WARN] Domain model richness (dom_003)
    Entities and enums are Lombok `@Data` data carriers with almost no behavior; business rules (distance to nearest happy location, grouping) live in the service and repository projections, which fits an anemic domain style rather than a rich domain model.
    • src/main/java/com/example/moodsense/demo/model/Mood.java
    • src/main/java/com/example/moodsense/demo/model/Location.java
    • src/main/java/com/example/moodsense/demo/service/MoodServiceImplementation.java
  • [WARN] Value objects (dom_005)
    `Coordinate` is a Spring Data projection interface (getters only), not an explicit immutable value object (for example a `record` or small final class with validation). It works for query mapping but does not embody a dedicated value-object boundary in the domain.
    • src/main/java/com/example/moodsense/demo/model/Coordinate.java
    • src/main/java/com/example/moodsense/demo/repositary/MoodRepository.java
  • [WARN] Naming conventions (dom_008)
    Most names follow Java conventions; minor issues include `getHappyNearByLocations` (awkward `NearBy` segment in camelCase) and the verbose `MoodServiceImplementation` name. The misspelled package name is covered under package naming.
    • src/main/java/com/example/moodsense/demo/service/MoodService.java
    • src/main/java/com/example/moodsense/demo/controller/MoodSenseController.java
  • [WARN] Single Responsibility (dom_009)
    `MoodServiceImplementation` both delegates persistence and implements location resolution and Euclidean distance logic (`getBestLocationFromCoordinate`), mixing application orchestration with spatial matching concerns that could live in a dedicated component.
    • src/main/java/com/example/moodsense/demo/service/MoodServiceImplementation.java
  • [PASS] Separation of concerns (dom_002)
    The codebase uses distinct packages for controller, service, persistence, and model types, and the controller delegates to `MoodService` rather than calling the repository directly.
  • [PASS] Interface-driven design (dom_007)
    `MoodService` is a dedicated interface with a Spring `@Service` implementation registered as `moodService`, which satisfies interface-driven service design.

§3. Business Logic (0/10)

Perspective: Transactional integrity, async messaging reliability, resilience patterns, maintainability, idempotency, and domain rule enforcement.

  • [FAIL] Logic correctness (biz_001)
    Nearest-location distance in getBestLocationFromCoordinate is wrong: the second squared term uses location.getLatitude() minus coordinate.getLongitude() instead of location.getLongitude() minus coordinate.getLongitude(). That breaks the intended Euclidean distance in the lat/long plane and can pick the wrong Location.
    • src/main/java/com/example/moodsense/demo/service/MoodServiceImplementation.java
  • [FAIL] Transaction management (biz_002)
    MoodServiceImplementation.save delegates to the repository but neither the service method nor the interface is annotated with @Transactional. For consistency and clear transactional boundaries on writes (rollback rules, propagation), the save path should declare an explicit transaction at the service layer.
    • src/main/java/com/example/moodsense/demo/service/MoodServiceImplementation.java
    • src/main/java/com/example/moodsense/demo/service/MoodService.java
  • [FAIL] Null safety (biz_004)
    getHappyNearByLocations and getLocationsFromCoordinates do not guard against a null remoteUser or null coordinate list. getBestLocationFromCoordinate returns null when there are no Location rows (or no improvement), but getLocationsFromCoordinates still adds that null to the result list, which can surface null elements to API clients.
    • src/main/java/com/example/moodsense/demo/service/MoodServiceImplementation.java
  • [FAIL] Performance - N+1 queries (biz_005)
    getBestLocationFromCoordinate calls moodRepository.getAllLocations() inside its loop, and getLocationsFromCoordinates invokes getBestLocationFromCoordinate once per happy coordinate. That reloads all locations from the database for every coordinate instead of loading the location list once and reusing it.
    • src/main/java/com/example/moodsense/demo/service/MoodServiceImplementation.java
  • [FAIL] Method visibility (biz_008)
    Helper methods getLocationsFromCoordinates and getBestLocationFromCoordinate are public though they are only used internally to MoodServiceImplementation. They should be private (or package-private) to avoid expanding the service API surface.
    • src/main/java/com/example/moodsense/demo/service/MoodServiceImplementation.java
  • [FAIL] Repository ID type mismatch (biz_009)
    MoodRepository extends CrudRepository but Mood declares @Id private int id. Spring Data expects the second generic to match the entity identifier type; using String here is inconsistent and risks incorrect generated API usage (findById, deleteById, etc.).
    • src/main/java/com/example/moodsense/demo/repositary/MoodRepository.java
    • src/main/java/com/example/moodsense/demo/model/Mood.java
  • [WARN] Service layer completeness (biz_003)
    uploadMood in the controller assembles the Mood (person from request, coordinates, status) before calling moodService.save. That orchestration is thin but still couples HTTP concerns to entity construction; a dedicated service method (e.g. recordMood) would keep the controller as transport-only and centralize invariants.
    • src/main/java/com/example/moodsense/demo/controller/MoodSenseController.java
    • src/main/java/com/example/moodsense/demo/service/MoodServiceImplementation.java
  • [WARN] Magic values (biz_006)
    getAllHappyCoordinates uses the string literal 'HAPPY' in JPQL. If MoodStatus enum values change, the query can drift. Prefer a bind parameter or a form that references the enum consistently (e.g. :status) supplied from application code.
    • src/main/java/com/example/moodsense/demo/repositary/MoodRepository.java
  • [WARN] Immutability and side-effects (biz_007)
    getHappyNearByLocations appears to be a pure read but triggers repeated repository reads (getAllLocations per coordinate), making behavior depend on hidden I/O and load. Predictable, side-effect-light methods would fetch data once and compute in memory.
    • src/main/java/com/example/moodsense/demo/service/MoodServiceImplementation.java
  • [WARN] Input sanitization (biz_010)
    uploadMood validates moodStatus, latitude, and longitude with @Validated @RequestParam, and Mood has JPA validation annotations, but the Mood binding object is not @Validated on the method, request.getRemoteUser() is not validated or normalized, and there is no explicit range/domain checks in the service beyond what the controller passes through.
    • src/main/java/com/example/moodsense/demo/controller/MoodSenseController.java
    • src/main/java/com/example/moodsense/demo/model/Mood.java

§4. Data Access (1/10)

Perspective: Repository patterns, transactional integrity, connection pooling, schema migrations, query efficiency, caching, and integration performance.

  • [FAIL] Entity ID type consistency (da_002)
    Mood declares @Id @GeneratedValue on a primitive int field, but MoodRepository is parameterized as CrudRepository. The repository ID type must match the entity identifier type (here int/Integer, not String). This mismatch can break type-safe repository methods (findById, deleteById) and indicates a serious mapping error.
    • src/main/java/com/example/moodsense/demo/repositary/MoodRepository.java
    • src/main/java/com/example/moodsense/demo/model/Mood.java
  • [FAIL] N+1 query problem (da_004)
    getBestLocationFromCoordinate calls moodRepository.getAllLocations() inside a for loop over coordinates (via getLocationsFromCoordinates). Each coordinate iteration re-executes the full locations query, causing N repeated identical reads instead of loading locations once and reusing the list in memory or pushing distance logic into the database.
    • src/main/java/com/example/moodsense/demo/service/MoodServiceImplementation.java
    • src/main/java/com/example/moodsense/demo/repositary/MoodRepository.java
  • [WARN] Repository extends appropriate interface (da_001)
    MoodRepository extends CrudRepository, which is valid but less idiomatic than JpaRepository for a Spring Data JPA application. JpaRepository adds useful defaults (e.g. flush, batch delete helpers, List return types on findAll) without changing behavior much. The larger problem is the incorrect ID generic (see da_002), not the choice between Crud and Jpa per se.
    • src/main/java/com/example/moodsense/demo/repositary/MoodRepository.java
  • [WARN] Use of production-grade database (da_003)
    application.properties configures jdbc:h2:mem:testdb with H2Dialect and enables the H2 console. In-memory H2 is appropriate for demos, local development, and tests, but it is not suitable for production persistence (no durability across restarts, limited operational tooling). For production you would profile a managed RDBMS and use non-memory URLs or pooled external H2 only where justified.
    • src/main/resources/application.properties
    • pom.xml
  • [WARN] Query hardcoded enum string (da_005)
    getAllHappyCoordinates uses JPQL with moodStatus = 'HAPPY' as a literal. Prefer a bind parameter (e.g. :status) typed as MoodStatus or use a derived query so renames and enum changes are compile-time checked and the value is not duplicated as a magic string.
    • src/main/java/com/example/moodsense/demo/repositary/MoodRepository.java
  • [WARN] Proper use of @Transactional (da_006)
    No @Transactional annotations appear on the service layer. Spring Data repository methods are transactional by default, so saves and queries will still run in transactions, but service-level boundaries are implicit. For clearer semantics and optimization, read-only service methods could use @Transactional(readOnly = true), and multi-step workflows could use an explicit propagation strategy.
    • src/main/java/com/example/moodsense/demo/service/MoodServiceImplementation.java
  • [WARN] Database migration tool (da_008)
    pom.xml has no Flyway or Liquibase dependency; schema is effectively managed by JPA ddl-auto defaults (not explicitly set in application.properties). For production, versioned migrations are recommended to make schema changes repeatable and reviewable.
    • pom.xml
    • src/main/resources/application.properties
  • [WARN] Connection pool configuration (da_009)
    No explicit HikariCP (or other pool) settings are present. Spring Boot 2.7 still defaults to HikariCP on the classpath, so pooling works out of the box, but production workloads typically tune maximum pool size, connection timeout, and leak detection via spring.datasource.hikari.*.
    • src/main/resources/application.properties
    • pom.xml
  • [WARN] Repository naming (da_010)
    The Java package is named com.example.moodsense.demo.repositary, which misspells repository. This hurts discoverability and consistency with conventional naming (repository).
    • src/main/java/com/example/moodsense/demo/repositary/MoodRepository.java
    • src/main/java/com/example/moodsense/demo/service/MoodServiceImplementation.java
    • src/test/java/com/example/moodsense/demo/service/MoodServiceImplementationTest.java
  • [PASS] Fetch strategy (da_007)
    Entities Mood and Location are standalone with scalar fields only; there are no @OneToMany, @ManyToOne, or other associations that require an explicit fetch type (LAZY vs EAGER). No fetch strategy misconfiguration applies given the current model.

    The Mood entity declares 'latitude' and 'longitude' as primitive 'int' fields, then annotates them with @NotNull.
